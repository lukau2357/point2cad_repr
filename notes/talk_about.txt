---- Post Point2CAD ----
- CADRecode simply did not work for 2 sample point clouds. It has a context length of 256, single ABC clusters contain up to 1000 points, 
possibly even more (freeform surface sample point cloud, the freeform cluster contains 7.2k points). Furthermore, the dataset used for trainig 
was procedurally generated - CADQuery scripts, and as such limits to CAD models that can be represented as a sequence of 
sketch/extrude operations, limiting bias for industrial point clouds. We examine this by passing an ABC CAD model - point cloud downsampled
to 256, the resulting CADQuery model is nowhere neal the original point cloud (demonstration in Kaggle notebook).

- Point2Primitive is similar, in a sense that it represents a CAD model as a sequence of sketch/extrude operations. This model 
uses the DeepCAD dataset, which is fundamentally different from the ABC dataset because the ABC CAD models are not annotated with
sketch/extruce operations. Again, suffers from "simplicity bias" - CAD models representable as sequences of sketch/extrude operations.
Uses a transformer neural network, so the same flaw as CADRecode - limited context length, 2048 in this case.

- P2CadNet: Trained on the same datasret as Point2Primitive, also represents a CAD model as a sequence of sketch/extrude commands,
also uses tranformers and as such suffers from limited context length.

Thoughts on these models: DeepCAD inherently limits the expressive power of a model if we take the deep-learning centric approach - 
complexity of models is limited to sketch/extrude sequences. Furthermore, the usage of transformers puts an upper bound on the resolution
of the input point cloud - both properties are not desired in practice./

Idea: Extend Point2CAD primitive surface fitting to facilitate a full BRep model, rather than a discrete mesh approximation.
